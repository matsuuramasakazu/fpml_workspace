# 5. Notional Amortization Resolution for Calculation Periods

## Status

Accepted

## Context

想定元本のステップスケジュール（`notionalStepSchedule`）を持つ元本削減・変動（Amortization/Step-up）スワップにおいて、各計算期間（`CalculationPeriod`）に割り当てる想定元本（`notionalAmount`）の解決方法について議論となりました。

特に、計算期間の途中で元本が変動するような設定（`stepDate` が計算期間の期中にある）の場合に、「期間内の加重平均元本」を算出して割り当てるべきか、あるいは「開始日時点の元本」をそのまま割り当てるべきかが論点となりました。

## Decision

各計算期間の **「調整前開始日（`unadjusted_start_date`）」** を基準日とし、その時点の想定元本をそのまま割り当てる方針を採用しました。

理由は以下の通りです：
- **スキーマ上の制約**: FpMLの `CalculationPeriod` 複合型における `notionalAmount` 要素は単一（`maxOccurs="1"`）であり、1つの計算期間内で複数の元本を持つことはデータ構造上想定されていません。
- **ISDA/FpMLの設計原則**: もし計算期間の途中で元本が変動するような特殊な取引を処理する場合、加重平均などの近似値を計算するのではなく、元本変動日（`stepDate`）を境界として計算期間（`CalculationPeriod`）自体を分割（サブ期間化）してそれぞれに元本を割り当てるのが、FpMLおよびISDAの正確な表現です。
- **標準的な契約設計**: 通常の金利スワップ（IRS）や通貨スワップ（CCS）の契約においては、元本変動日（`stepDate`）は必ず計算期間の開始日（`unadjusted_start_date`）と一致するように設計されます。
- **結論**: そのため、本ツールでは「`stepDate` は計算期間の開始日に必ず一致している」という実務上の前提を置き、`unadjusted_start_date` 時点での単純なルックアップを行うのが最も正確かつ標準的であると判断しました。

## Consequences

### Positive (良い影響)
- FpMLおよびISDAのデータ構造・設計原則に完全準拠します。
- 不必要な近似計算ロジック（加重平均等）を排除することで、コードがシンプルになり、不整合バグを回避できます。

### Negative (懸念される影響)
- 万が一、計算期間の途中で元本がステップするような特殊な契約データ（例：特殊なローン連動スワップ）が入力された場合、本ツールは期間分割を行わず、開始日時点の元本を割り当てるため、意図しない金額が出力される可能性があります。そのような特殊取引に対応する場合は、将来的に「元本変動日でのスケジュール自動分割ロジック」を `CalculationPeriodScheduler` に実装する必要があります。
