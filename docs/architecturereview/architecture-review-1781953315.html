Problem (問題点)

CalculationPeriodScheduler と PaymentPeriodScheduler の両方に、期間生成と月数加算のロジック (_generate_unadjusted_dates や _add_months) がほぼ同一のコードで重複定義されており、局所性 (locality) が損なわれています。また、SwapStreamScheduler は単にこれら2つのスケジューラを仲介するだけの極めて浅い (shallow) モジュールとなっています。

Solution (解決策)

CalculationPeriodScheduler と PaymentPeriodScheduler の実装を SwapStreamScheduler 内にプライベートにマージし、重複していた日付生成・加算ロジックを一箇所に集約して完全に隠蔽します。外部に対する interface は SwapStreamScheduler.generate_payment_periods と generate_principal_exchanges のみに制限し、モジュールを深く (deep) します。

Wins (効果)

  • Locality: スケジュール計算と支払集約のロジックが1箇所に凝縮され、保守が容易になります。
  • Leverage: 外部のテストや呼出元は SwapStreamScheduler の単一の interface のみを意識すればよくなります。
  • Code reduction: 重複していた日付前進/後退生成ロジック (約100行) を安全に削除できます。

Problem (問題点)

DateAdjuster は BusinessCalendar と ReferenceResolver を内包し、FpML固有の調整設定型 (BusinessDayAdjustments や RelativeDateOffset) を受け取ってカレンダー処理に委譲するだけの非常に浅い (shallow) 仲介モジュールとなっています。このため、各 Scheduler は個別に DateAdjuster をインスタンス化して引き回す必要があり、余計な依存が生じています。

Solution (解決策)

DateAdjuster を削除し、その日付調整・解決ロジックを統合後の SwapStreamScheduler のプライベート実装（あるいはユーティリティヘルパー）として取り込みます。これによって、ドメイン固有の XSD オブジェクト解釈は SwapStreamScheduler 内に完全にカプセル化され、カレンダー (BusinessCalendar) には純粋な日付計算処理のみを依存させる形に整理されます。

Wins (効果)

  • Simpler seams: 各 Scheduler から DateAdjuster という余分な seam が排除され、直接 BusinessCalendar とやり取りするか、SwapStreamScheduler 内部で処理が完結します。
  • Locality: FpML 特有の XSD スキーマ構造を解釈するコードが SwapStreamScheduler 内部に集約され、インターフェース設計がすっきりします。

Top Recommendation (最優先推奨アクション)

候補 1: スケジュール生成モジュール群の統合と共通化 (Deepen Schedulers) を最優先で進めることを推奨します。

理由: 日付生成ロジックの完全な重複は、コードベース内で最も深刻な保守上の friction となっています。また、SwapStreamScheduler, CalculationPeriodScheduler, PaymentPeriodScheduler の関係は「同一のインプロセス機能（スケジュールの構築と展開）」に属しており、これらを一つの deep なモジュールに統合することで、テストのしやすさと実装の局所性が飛躍的に向上します。
